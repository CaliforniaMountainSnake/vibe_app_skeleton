## Code

- В проекте строгий ESLint с правилами на цикломатическую сложность, размеры функций и файлов, именование и стиль.
- Ошибки и предупреждения линтера - не suggestions, их нужно исправлять.
- Не выключай сообщения линтера, не переписывай его настройки без прямого указания.
- Если линтер говорит, что в файле/модуле/функции/etc слишком много строк - значит, нужно разбить его на меньшие для контроля сложности.
- Агент самостоятельно ничего не изменяет в системе контроля версий, не пушит, не пулит, не переключает ветки, максимум, можно что-то прочитать из истории, если нужно.

### Действия после внесения любых изменений в кодовую базу

- Удостовериться, что линтер (`npm run lint`) не выдает ошибок.
- Прогнать юнит-тесты (`npm run test`).
- Если линтер и тесты не выдают ошибок, в самом конце выполнить форматирование исходных кодов (`npm run format`).
- После каждой итерации изменений, независимо от их масштаба, выводи: название ветки в формате `Conventional Commits` (предварительно проверь, что она ещё не занята) и текст коммита на русском языке, объясняющий зачем были внесены все накопленные за сессию изменения (общая цель, без технических деталей).

## UI

- В проекте используется `shadcn/ui` как готовая библиотека компонентов.
- Ты НЕ имеешь права переписывать или переизобретать её компоненты.
- Не создавай свои версии компонентов из `shadcn/ui`.
- Если компонент уже есть в `shadcn/ui` — используй его.
- Не копируй код компонентов в новые файлы.
- Не заменяй shadcn компоненты кастомным Tailwind-кодом.
- Не делай "улучшенные" абстракции поверх них.
- Не изменяй имеющиеся компоненты без прямого указания.
- В папках с компонентами должны лежать только ui-компоненты, код, который не относится к ui, выноси в хелперы/утилиты и т.п.
